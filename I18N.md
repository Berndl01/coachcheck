# Zweisprachigkeit (DE/EN) — Stand & Fortführung

Entscheidung: Englisch **zusätzlich** zu Deutsch (Sprachumschalter), Deutsch bleibt
Standard und vollständig erhalten. Kein `[locale]`-Routen-Umbau — Cookie-basiertes i18n.

## Fundament (Welle 1, v3_57)
- `lib/i18n/config.ts` — Sprachen `['de','en']`, Standard **de**, Cookie `cc_locale`.
- `lib/i18n/dictionaries/{de,en}.ts` — EN-Struktur per `Dictionary`-Typ erzwungen identisch zu DE.
- `lib/i18n/index.ts` — `makeT(locale)`: Punktpfad-Lookup, Fallback DE -> Key.
- `lib/i18n/server.ts` — `getLocale()` / `getT()` (Server-Komponenten).
- `components/i18n/locale-provider.tsx` — `LocaleProvider` + `useT()` (Client).
- `components/i18n/language-toggle.tsx` — DE/EN-Umschalter (Cookie + Reload).
- `app/layout.tsx` — `<html lang>` dynamisch, Provider, lokalisierte Metadaten.

## Landing vollständig zweisprachig (Welle 2, v3_58)
Alle 17 Landing-Komponenten umgestellt — Klick auf EN macht die **gesamte Startseite**
englisch (Nav, Hero, Stats, Ticker, Problem, Pakete, Archetypen, Mini-Check, Architektur,
Wissenschaft, Ablauf, Beispiel-Report, Stimmen, Quote, Trust-Band, FAQ, Final-CTA, Footer).
19 Wörterbuch-Sektionen, DE/EN strukturgleich.

Bewusste Festlegungen (gehalten & geprüft):
- Produktnamen (Schnelltest, Selbsttest, 360° Spiegel, TeamCheck, Saison) bleiben Deutsch —
  kanonisch, deckungsgleich mit DB/Checkout/Dashboard/PDF.
- 12 Archetyp-Namen bleiben Deutsch (psychometrischer Modellinhalt); nur Eigenschafts-
  Stichworte + Marketing-Rahmen übersetzt.
- Akademische Zitate (Autoren/Jahr) bleiben; nur Themenbeschreibung übersetzt.
- Rechtsseiten-Labels (Imprint/Privacy/Terms) übersetzt, Seiten bleiben Deutsch (Anwalt-Spur).
- FAQ-Markenkorrektur: "Mindset Check" -> "CoachCheck" (DE + EN).
Echte EN-Produktnamen = spätere `name_en`-DB-Spalte (App-Welle).

## App-Seiten Teil 1: Käufer-Reise (Welle 3a, v3_59)

Globales Chrome + Einstieg + Transaktion + Konto sind zweisprachig (Formulare/Chrome,
kein psychometrischer Inhalt). 14 Dateien:
- Chrome: `cookie-banner` (Client; schließt zugleich eine Landing-Lücke, da im Root-Layout),
  `error.tsx` (Client), `not-found.tsx` (jetzt async Server).
- Auth: `login`, `signup`, `passwort-vergessen`, `passwort-neu` (alle Client; Fehlermeldungen
  in Handlern/Effects ebenfalls auf `t()`).
- Checkout: `checkout/[slug]` (Server; sprachabhängige Währung de-DE/en-IE), `consent-form`
  (Client), `checkout/success` (Server).
- Konto: `konto/daten` + `data-controls` (Client), `profil/setup` + `setup-form` (Client;
  Niveau/Alter/Vereinstyp-Labels übersetzt, DB-Schlüssel bleiben).

Festlegungen:
- DB-Auswahlwerte (sport=fussball, role=trainer, training_level=leistungszentrum …) bleiben;
  nur sichtbare Labels übersetzt → eigene `options`-Sektion.
- **FLAG (Recht/Anwalt):** Die `consent-form` enthält rechtlich operativen Text (FAGG-
  Widerrufsverzicht, DSGVO-Einwilligung). Deutsch ist die **operative** Fassung; die englische
  ist eine **Verständnishilfe (Entwurf)** und muss vor dem EN-Markt anwaltlich freigegeben
  werden — wie die Rechtsseiten. (Code-Kommentar in consent-form vermerkt das.)
- **FLAG (DB):** Checkout zeigt `product.name_de` + `product.description` aus der DB (Deutsch).
  Echte EN-Produktnamen/-beschreibungen = spätere Spalten `name_en`/`description_en`.
- Korrektur: Consent-Hinweis sagte „alle drei Punkte", es sind aber **vier** Checkboxen → „vier".

## Muster (Referenz)
Server: `const t = await getT();` (`@/lib/i18n/server`) -> `{t('section.key')}`.
Client: `'use client'` + `const t = useT();` (`@/components/i18n/locale-provider`).
Wörterbuch: `de.ts` führend; `en.ts` muss dieselben Keys haben (Typ erzwingt es).
Platzhalter: `t('key').replace('{name}', wert)`.
